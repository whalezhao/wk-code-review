## 1 解决的核心问题
- 瀑布模式内容过多，测试太慢，影响后续迭代进度
- 需求描述模糊，覆盖范围特别大，导致协作混乱
- 多个版本并行开发，如何互不干扰？
- 以更现代的方式，进行代码审查，提高审查效率
- 减少沟通成本：
  - 某个分支开发的需求是是什么？
  - 这个迭代，涉及哪些分支？
  - 某个分支属于哪个迭代？
  - 某个分支是谁负责？
  - 本次迭代所涉及的分支是否都已合并？
- 如何高效地将需求提前或延迟发布，并不会造成任何影响？
  - 进行中的需求提前或延后
  - 已完成的需求提前或延后
  - 下一个版本的需求提前
- 如何实现CI-Friendly？即，版本号自动识别而非硬编码
- 如何确保开发人员遵守了GIT管理规范？

## 2 理念
### 2.1 多版本并行开发
多个版本迭代的需求，可以并行开发，互不影响其进度。  
多个版本的最新特性，可以在开发环境及时呈现。

### 2.2 基于需求驱动的敏捷开发
**精细化管理**  
产品侧需要对需求精细化管理，将一个超大需求，拆分成最细颗粒度的需求，并按照优先级进行迭代。 
通过完成一个个的子需求，将大型需求逐步完成。   

**关联分支**    
- 每个 `feature` 分支严格对应一个需求，一个需求可以有多个 `feature` 分支

如，迭代1.0.0中的需求#123，由clouds3n和other-developer联合开发：  
```
feature/1.0.0/clouds3n/#123/common
feature/1.0.0/clouds3n/#123/function1
feature/1.0.0/clouds3n/#123/function2
feature/1.0.0/other-developer/#123/function3
```

- 每个 `bugfix` 分支严格对应一个缺陷
- 使用项目管理平台的ID，进行跟踪与关联，确保开发和测试围绕需求展开。  

**隔离开发**  
每个需求的开发、测试、发布是独立进行的，不受其他需求的进度影响，确保需求的开发和测试是滚动进行的。

### 2.3 高效的代码审查
审查范围明确，每个 feature 分支只包含本次需求的变动，代码审查集中在当前提交的改动，减少其他不相关部分的干扰。  
通过自动化CI流程，进行规范校验和测试，减轻审查负担，提供可量化的审查指标。  

### 2.4 灵活的需求调度
小步快走，避免臃肿的迭代，按照迭代中需求的优先级顺序开发，截止到deadline时，完成多少需求就发多少需求，防止版本持续延期。

可以更方便的将某个需求从迭代中剔除，支撑需求提前或延迟发布。  

### 2.5 确保仓库安全
根据最小权限原则，细化权限管理，保护核心分支和标签。  
只有特定的管理员和负责人才能审查和合并代码到核心分支。  

### 2.6 自动化流程保障
分支的合并、测试、发布等，增强自动化，减少人工干预。  

## 3 分支管理模型

![分支管理模型](/images/git-specification/git提交流程.png)  

以需求为最小单元进行开发。  
- [SNAPSHOT] 迭代开始，从master创建 `qa/x.x.x` 和 `release/x.x.x`分支
- [SNAPSHOT] 开发人员收到需求或缺陷，从master创建 `feature/x.x.x/username/<issure id>`
- [SNAPSHOT] 开发人员在 `feature` 分支上 `commit` 业务代码，并 `自测`
- [SNAPSHOT] 自测通过，将feature分支 `merge` 到 `nightly`分支，并 `联调`
- [SNAPSHOT] 联调通过，创建 feature 到 `qa` 分支的MR，进行 `提测`
- [SNAPSHOT] 测试过程中，`本次迭代涉及的缺陷` ，都在对应需求的 `feature` 上进行，再MR到qa分支
- [SNAPSHOT] 三轮测试完毕，产品确定可以发布的需求列表，开发将相关需求的 `feature` 分支，`squash` 为一个commit
- [SNAPSHOT] 创建 feature 到 `release` 分支的MR，部署 `预生产环境` 进行最后一轮 `回归测试`
- [正式版本] 测试完毕，release分支打 `x.x.x` tag，根据tag进行构建，发布生产环境
- 清理所有完成的分支

---

| 来源分支                | 目标分支  | 合并策略                           | 合并目的                                                                                                      |
| :---------------------- | :-------- | :--------------------------------- | :------------------------------------------------------------------------------------------------------------ |
| `feature`               | `nightly` | `git merge`                        | **部署开发环境联调**，将需求代码集成到公共联调分支，以验证与前端或其他后端模块的正确性。                                  |
| `feature`               | `qa`      | **创建 MR**（`git merge`）         | **部署测试环境提测**，将功能代码提交给测试团队，进行功能和回归测试。                                                      |
| `feature` (已 `squash`) | `release` | **`Squash and merge`** 策略        | **部署预生产环境**和**回归测试**。将一个需求的完整改动压缩成一个干净的提交，保持 `release` 分支提交历史的整洁。 |
| `release`               | `master`  | **`Fast-forward only merge`** 策略 | **正式发布**。将已验证和发布的版本代码同步到 `master` 分支，保持 `master` 提交历史的线性和整洁。              |

## 4 提交信息规范
> 遵循[Angular提交信息规范](https://github.com/angular/angular/blob/22b96b9/CONTRIBUTING.md#-commit-message-guidelines)

**提交格式**  
> 提交信息必须遵循此格式。  

每次提交可以包含页眉(header)、正文(body)和页脚(footer)，每次提交必须包含页眉内容。  
```
<type>(<scope>): <subject>
<BLANK LINE>
<body>
<BLANK LINE>
<footer>
```

**提交类型 type**  
允许使用以下类型：  
- `build`：对构建系统或者外部依赖项进行了修改
- `ci`：对CI配置文件或脚本进行了修改
- `docs`：对文档进行了修改
- `feat`：增加新的特征
- `fix`：修复bug
- `perf`：提高性能的代码更改
- `refactor`：既不是修复bug也不是添加特征的代码重构
- `style`：不影响代码含义的修改，比如空格、格式化、缺失的分号等
- `test`：增加确实的测试或者矫正已存在的测试

**作用域 scope**  

该约定确保功能与缺陷的变更可直接追溯到具体的任务或问题单，便于代码审查与发布管理。  

**分支命名与scope关联**
- 分支命名格式：`feature/<版本>/<开发者>/<需求ID>` 或 `bugfix/<版本>/<开发者>/<缺陷ID>`
- 分支名称最后的号码即为规范要求的scope号码
- 示例：分支 `feature/1.4.0/赵寒/489822` 对应的scope为 `#489822`

强制要求：  
- 当提交类型为 `feat` 或 `fix` 时，作用域（scope）必须对应项目管理系统中的需求或缺陷编号。
- 格式统一为： `#<ID>`，例如：
  - feat(#11241): 新增用户注册功能
  - fix(#9939): 修复登录接口参数校验错误

其他类型提交（如 docs、chore、refactor、test 等），作用域可选，建议使用分支对应的需求ID或简要描述修改影响的模块，例如：
- docs(readme): 更新使用说明
- refactor(#489822): 重构xxx代码
- refactor(auth): 优化认证逻辑结构
- build: 增加xxx依赖


**主题 subject**  
主题包括了对本次修改的简洁描述，有以下准则：  
- 一句话简单描述
- 使用命令式，现在时态：“改变”不是“已改变”也不是“改变了”
- 不在末尾添加句号

**正文 body**  
详细描述包含修改的动机，修改内容较多时，必须以清单方式列出具体的修改内容。如：  
```
feat(#123): 优化主页内容展示

- 优化用户头像圆框，更顺滑
- 优化个人任务提示方式
- 优化xxxx
```

**页脚 footer**  
页脚主要用于强调是破坏性改动（Breaking changes），或者与项目管理系统进行交互。  
破坏性修改，以 `BREAKING CHANGE:` 开头，后跟一个空格或者两个换行符，其余的信息就是对此次修改的描述和注意事项  
```
BREAKING CHANGE:

任务隔离策略进行了重构，需要对历史数据进行迁移操作。
迁移数据注意事项：
......
```

如果项目管理平台集成了代码仓库，则可以通过 issue ID快速更改任务状态：  
```
Closes #123, #245, #992
```

**合规案例**  

```
feat(#123): 添加xxx功能
fix(#321): 修复xxxbug
style: 调整代码格式
docs: 添加代码贡献说明
doc(#789): 更新xxx注释
build: 更新xxx依赖
build(#789): 更新xxx脚本
refactor(#456): 重构xxx代码
refactor(web): 重构AuthFilter支持白名单
```

## 5 MR规范

> 新项目的根目录，都会有MR申请模板 `.gitlab/merge_request_templates/default.md`  

- 开发的过程中，可以持续提交 `Draft:` 类型的MR，让负责人提前了解开发方向是否正确。  
![mr-draft](/images/git-specification/mr-draft.png)  

- 联调通过，并且需求的冒烟测试完成，才可以发起正式MR
![mr-requirements](/images/workflow/需求关联冒烟测试.png)

- 必须将本地分支的所有commit，squash为一个后，再提交最终的MR

- MR的title格式要与 `squash` 后的commit信息一致  
 
- MR提交时，选择 `default` 模板，并按照里面的要求，完成自检  

![mr-tempalte](/images/git-specification/mr-template.png)  

- pipeline全部成功后，再通知审核人走查代码

- 代码审核人会给开发人员整改建议，开发人员需要全部解决：  
![mr-comment](/images/git-specification/mr-comment.png)  

## 6 关于热修

- 开发过程中，只有现场**有阻塞**的功能或缺陷才能作为热修发布
- 注意热修分支采用截止时间为分支名，如，hotfix/2.8.x-0325
  - 代表这是2.8版本的热修，上线时间为3月25日
  - 该分支的pom version为 `2.8.x-0325-SNAPSHOT`
  - 3月25日发布该热修，打稳定版本时，才会决定最终的pom version，如，2.8.2
  - 这样做的目的是，防止过早定版，如果有更紧急的事项可以灵活调整

## 7 避免冲突

冲突是不可避免的，使用GIT就是要解决冲突，我们会从多个角度尽量避免冲突：  

1. 需求尽量独立，一个人负责一个需求的开发  

2. 同一个项目的同一个需求，涉及多个人开发，会先拉 `feature/2.x.x/common/需求id` 分支，  
然后大家在上面维护自己的接口、公共枚举、公共dto、orm层的curd等共性的代码，  
然后从这个分支拉取自己的私人分支，完成接口的实现，  
中途任何common分支的修改，都要及时同步  

3. 不兼容升级的，不能多个迭代并行开发

4. 涉及大范围重构、重写的，不能多个迭代并行开发

## 8 配套脚本
为方便日常操作，项目提供以下辅助脚本：

| 脚本命令 | 描述 |
| :--- | :--- |
| `check_merge <version> <branch>` | 检查指定版本的 feature/bugfix 分支是否已合入目标分支 (release/qa)。 |
| `delete_local_branches` | 清理本地除 master/main 外的所有分支。 |
| `delete_merged_branches <version>` | 在 `release/<version>` 上识别并删除已合并的 feature 分支。 |
| `rename_git_branch <old> <new>` | 重命名本地分支并同步更新远程分支。 |
| `git_change_summary` | 生成相对 `master..HEAD` 的变更摘要 (自动排除 style/build/test 类提交)。 |
| `user_code <dir> '<authors>' <date>` | 统计指定作者从指定日期开始的代码增删改行数。 |
