# WK Code Review Skill

## 自动模式判断

**在执行任何确认步骤之前，先判断当前运行模式：**

- 读取最新一轮 system-reminder 中的 `auto mode is active` 标识
- 或检查当前对话上下文中是否有 `auto_mode`、`autonomous`、`continuous` 等关键字
- 若无法判断，按**手动模式（默认）**处理

- **Auto Mode**：跳过所有用户确认步骤，直接执行审计
  - 不输出确认提示，不等待用户输入
  - Phase 0 使用默认输出目录 `{OUTPUT_DIR}/` 如果用户没有输入，根据操作系统放到桌面上 文件夹名称为 code-review
  - Phase 1.5 直接检查并恢复/跳过
  - Phase 2.6 每批次完成后自动继续下一批次
- **手动模式（默认）**：严格按流程执行，所有确认点生效

> **Skill 调用者有责任在调用时声明运行模式（auto_mode=true/false）。**

---

## 目标

执行完整代码审计，覆盖六个维度：

1. **规范审计** — 命名、日志、异常、事务、MyBatis、Spring、并发、MySQL、前后端交互规范
2. **业务逻辑审计** — 验证业务规则实现、边界条件、错误处理完整性
3. **方案合理性审计** — 评估架构设计、事务边界、数据一致性、方案选择
4. **代码性能审计** — 评估代码性能、方案合理性、代码容错、安全兜底
5. **可靠性审计** — MQ消费幂等、定时任务防重、分布式锁、资源泄漏、Istio/K8s场景
6. **宕机恢复与逻辑接续审计** — 状态机完整性、可重入性、补偿机制、分布式事务边界（**新增重点维度**）

## 核心原则

> **⚠️ 强制要求：必须先生成审计计划文件，再决定是否执行审计。**
>
> 1. 审计计划文件必须实际写入磁盘 `{OUTPUT_DIR}/<branch>-plan.md`
> 2. 每个Agent读取自己的计划文件和规范文件执行审计
> 3. 每批次Agent完成后，更新主计划中的待合并结果文件列表
> 4. 最终汇总所有Agent结果到完整审计报告
>
> **架构限制**：本 Skill 仅支持单仓库场景。多仓库（monorepo 跨仓库或不同仓库的服务）需分别调用本 Skill，或使用 CI 层面的全局审计工具。

**⚠️ 必须在执行审计前确认输出目录，后续所有步骤均使用此目录。**

## Phase 0.5：预检查

执行以下命令获取上下文：

```bash
# 1. 获取当前分支（如果没有传入分支名，则取当前分支）
CURRENT_BRANCH=$(git rev-parse --abbrev-ref HEAD 2>/dev/null)
echo "当前分支: $CURRENT_BRANCH"

# 2. 获取基准分支（优先使用传入的 BASE_BRANCH，否则检测仓库默认分支）
BASE_BRANCH=${BASE_BRANCH:-$(git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's|refs/remotes/origin/||')}

# 3. 检查差异文件数
git diff --name-only ${BASE_BRANCH}...${BRANCH:-$CURRENT_BRANCH} 2>/dev/null | wc -l

# 4. 获取提交历史（理解需求背景）
git log ${BASE_BRANCH}..${BRANCH:-$CURRENT_BRANCH} --oneline -15

# 5. 获取变更统计
git diff --stat ${BASE_BRANCH}...${BRANCH:-$CURRENT_BRANCH}
```

> **无差异时**：提示"当前分支与 ${BASE_BRANCH} 没有差异"，终止审计。
>
> **参数说明**：
> - `${BRANCH}`：待审计分支名（可省略，默认当前分支）
> - `${BASE_BRANCH}`：基准分支名（可省略，自动检测仓库默认分支，通常为 master/main）

## Phase 1：生成主计划文件

### 1.1 统计变更文件

```bash
# 获取所有变更的Java和XML文件
# 注意：使用 ${BRANCH:-$CURRENT_BRANCH} 而非 HEAD，确保多批次执行时取到正确分支
git diff --name-only ${BASE_BRANCH}...${BRANCH:-$CURRENT_BRANCH} 2>/dev/null | grep -E '\.(java|xml)$'
```

### 1.2 筛选文件

#### 必须审计的文件

- 所有 `*.java` 文件（排除规则除外）
- 所有 `*Mapper.xml` 文件

#### 排除规则

| 模式 | 说明 | 是否计入审计 |
|---|---|---|
| `*Test.java` / `**/test/**` | 测试类 | ❌ 排除 |
| `**/generated/**` | 生成代码 | ❌ 排除 |
| `**/target/**` | 编译产物 | ❌ 排除 |
| `**/pom.xml` | Maven配置 | ❌ 排除 |
| `**/resources/META-INF/**` | Spring/META-INF配置 | ❌ 排除 |
| `**/resources/application*.yml` | 应用配置文件 | ❌ 排除 |
| `**/resources/application*.properties` | 应用配置文件 | ❌ 排除 |
| `**/resources/static/**` | 静态资源 | ❌ 排除 |
| `**/resources/templates/**` | 模板文件 | ❌ 排除 |
| `**/resources/i18n/**` | 国际化资源文件 | ❌ 排除 |
| `**/resources/mapper/**/*.xml` | MyBatis XML | ✅ **纳入审计**（检查SQL注入、拼接） |
| `**/resources/migration/**/*.sql` | 数据库迁移脚本 | ✅ **纳入审计**（检查SQL质量、索引） |

> **关于 YAML 配置文件**：如本次变更涉及 `application.yml` / `application.properties`，**不纳入审计范围**（已在排除规则中）。配置合理性（如连接池大小、超时配置）属于运维层面，不在代码审计维度内。

### 1.3 生成主计划文件

**输出位置**：`{OUTPUT_DIR}/<branch-name>-plan.md`

使用 Write 工具创建主计划文件，内容包括：

1. **统计信息表**
   - 变更文件总数、Java文件数、XML文件数、应审计文件数

2. **审计批次清单**
   - 按模块/层级分组，每个批次的文件列表

3. **执行步骤**（按步骤确认）

4. **待合并结果文件列表**
   - 初始为空，每批次完成后动态追加

5. **重点关注项**

### 1.4 为每个Agent生成独立计划文件

**输出位置**：`{OUTPUT_DIR}/<branch-name>-plan-<批次序号>-<批次名>.md`

> **注意**：批次名中不能包含 `/` 字符，避免文件系统路径问题。用 `-` 替代。

每个Agent计划文件必须包含：

1. **本次批次的文件列表**（完整路径）
2. **需求背景**（提交历史摘要）
3. **规范文件列表**（统一使用 `./references/` 相对路径）
4. **审计维度说明**
5. **结果文件路径**

### 1.5 输出计划摘要并等待确认（手动模式）

```
📋 审计计划已生成

分支：{branch-name}
变更文件：{total-files} 个
应审计：{to-audit-files} 个

审计批次：
  批次1：{name} ({n}个文件)
  批次2：{name} ({n}个文件)
  ...

执行步骤：
  Step 1: 批次1审计（启动{n1}个Agent）→ 更新待合并列表
  Step 2: 批次2审计（启动{n2}个Agent）→ 更新待合并列表
  Step 3: 汇总所有结果，生成最终报告
  Step 4: 输出摘要

计划文件：
  主计划：{OUTPUT_DIR}/{branch-name}-plan.md
  各批次：{OUTPUT_DIR}/{branch-name}-plan-*.md

请确认是否按此计划执行审计。
```

> **手动模式**：只有用户确认后，才能进入 Phase 2 执行审计。
> **Auto Mode**：跳过确认，直接进入 Phase 1.5。

## Phase 1.5：中断检查与恢复

在执行 Phase 2 前，检查是否有未完成的审计：

```bash
# 检查是否存在进行中的审计状态文件
STATUS_FILE="{OUTPUT_DIR}/${BRANCH}-status.json"
RESULT_FILES=$(ls {OUTPUT_DIR}/${BRANCH}-result-*.json 2>/dev/null)

if [ -f "$STATUS_FILE" ]; then
  echo "发现历史审计状态文件"
  cat "$STATUS_FILE"
fi

if [ -n "$RESULT_FILES" ]; then
  echo "发现历史结果文件:"
  echo "$RESULT_FILES"
fi
```

**恢复场景**：

1. **完全中断**：主计划存在但未任何 batch 完成
   - 手动模式：提示用户"发现未完成的审计计划，是否继续？"
   - Auto Mode：自动从 Phase 1.5 继续，从第一个未完成的批次开始

2. **部分完成**：某些 batch 已完成，但不是全部
   - 跳过已完成的批次
   - 从第一个未完成的批次继续执行

3. **新审计**：无任何历史审计文件
   - 正常开始新的审计流程

## Phase 2：执行审计（按步骤执行）

### Step 1-N：批次审计

每批次执行以下步骤：

#### 2.1 检查进度

```bash
# 确认本批次尚未完成
BATCH_RESULT="{OUTPUT_DIR}/${BRANCH}-result-${BATCH_ID}-merged.json"
if [ -f "$BATCH_RESULT" ]; then
  echo "批次 ${BATCH_ID} 已完成，跳过"
  exit 0
fi
```

#### 2.2 创建进度检查点

写入进度文件：`{OUTPUT_DIR}/${BRANCH}-status.json`

```json
{
  "branch": "{branch-name}",
  "timestamp": "{current-time}",
  "phase": 2,
  "current_batch": {n},
  "total_batches": {total},
  "completed_batches": [{已完成批次}],
  "pending_batches": [{待执行批次}],
  "in_progress": true
}
```

#### 2.3 启动Agent（每批次最多4个并行）

每个Agent执行：

1. Read 计划文件
2. Read 规范文件
3. Read 每个待审计文件的完整内容
4. Bash git diff 获取代码变更
5. **逐维度逐文件分析**，对每个维度明确输出结论（有问题/无问题/不适用）
6. 将结果写入JSON临时文件

**⚠️ 逐维度强制要求**：

对批次的每个文件，必须按以下顺序逐维声明：

| 维度 | 必须声明 |
|------|---------|
| A. 规范审计 | 有/无问题（明确标注：SQL注入/异常/日志/事务/并发各几项） |
| B. 业务逻辑 | 有/无问题（明确标注：需求覆盖/边界/null各几项） |
| C. 方案合理性 | 有/无问题（明确标注：N+1/重复代码各几项） |
| D. 性能 | 有/无问题（明确标注：分页/批量/异步各几项） |
| E. 可靠性（Istio/K8s） | 有/无问题（明确标注：幂等/状态外化/事务/锁各几项） |
| F. 宕机恢复与逻辑接续 | 有/无问题（**重点审计**，明确标注：状态机完整性/可重入性/补偿机制/分布式事务各几项） |
| G. 安全审计 | 有/无问题（明确标注：注入/敏感信息/权限各几项） |

**⚠️ 不得跳维度**：每个文件必须对 A-G 七个维度逐一声明结论，不能只输出有问题的维度。

> **⚠️ F维度特别提醒**：宕机恢复审计是本次新增的重点维度。审计时必须追问：
> - "如果在这行代码之后JVM宕机，会发生什么？"
> - "重启后这条消息会被重新投递吗？业务逻辑能否正确处理？"
> - "跨系统的两阶段操作（DB+MQ、DB+OSS、MQ+外部服务）是否有补偿机制？"

**Agent异常处理**：
   - Agent 应在执行过程中每处理 5 个文件后保存一次中间结果
   - 如果 Agent 崩溃，下次恢复时应检查中间结果并从断点继续
   - 使用结果文件命名约定：`result-<batch>-<agent-id>-checkpoint-{n}.json`

#### 2.4 监控进度

- 使用 TaskList 查看任务状态
- **超时时间动态计算**：`超时(min) = max(5, min(文件数 / 2, 20))`（最少5分钟，最多20分钟）
- 每 1 min 检查一次是否有 Agent 超时或异常

#### 2.5 合并批次结果

本批次所有 Agent 完成后：

1. **批次内去重**（同一 Agent 内）
   - 按 `文件路径 + 行号 + 维度 + 问题类型` 去重
   - 保留严重程度最高的问题（CRITICAL > WARNING > SUGGESTION）

2. **批次间去重**（同一批次不同 Agent 间）
   - 读取该批次所有 Agent 的结果文件
   - 按 `文件路径 + 行号 + 维度 + 问题类型` 去重
   - 同一个问题多个 Agent 发现时，保留 CRITICAL > WARNING > SUGGESTION

3. **合并到批次结果文件**
   - 写入 `{OUTPUT_DIR}/${BRANCH}-result-${BATCH_ID}-merged.json`

4. **更新进度检查点**
   ```json
   {
     "...": "...",
     "completed_batches": [..., {n}],
     "pending_batches": [{剩余批次}],
     "last_completed_at": "{timestamp}"
   }
   ```

> **⚠️ 注意**：跨批次的全局去重统一在 Phase 3 执行，此处仅处理批次内和批次间去重。

#### 2.6 更新主计划中的待合并列表

每批次完成并合并后：

1. 更新主计划的待合并列表
2. 追加本批次的合并结果文件路径
3. 标记该批次为 ✅ 完成

**手动模式**：输出确认提示
```
✅ 批次{n}审计完成（共 {file-count} 个文件，{problem-count} 个问题）

本批次结果文件：
  - {OUTPUT_DIR}/{batch-name}-merged.json

已更新主计划待合并列表。
共 {total-n} 个结果文件待汇总。

是否继续下一批次？
```

**Auto Mode**：自动继续下一批次，不等待确认。

#### 2.7 异常恢复策略

**场景1：单个Agent失败**
- 记录失败 Agent 的文件列表
- 启动备用 Agent 重新审计这些文件
- 最多重试 2 次

**场景2：整批次超时**
- 将该批次分解为更小的子批次（每个 3 文件）
- 重新启动，每个子批次用 1 个 Agent

**场景3：主进程中断**
- 下次恢复时自动检测进度检查点
- 从上次未完成的批次开始

**场景4：Agent 返回空结果**
- Agent 返回了结果但问题列表为空，视为异常
- 检查 Agent 是否正确读取了所有文件内容
- 启动备用 Agent 重新审计该批次所有文件
- 如果连续2次空结果，则报告该批次审计异常，需人工介入

#### 2.8 循环执行直到所有批次完成

重复 Step 1-N 直到所有批次完成

更新进度检查点为：
```json
{
  "...": "...",
  "phase": 3,
  "in_progress": false,
  "completed_at": "{timestamp}"
}
```

## Phase 3：汇总结果

### 3.1 读取所有待合并结果文件

根据主计划中的待合并列表，读取所有JSON结果文件（`*-result-{batch}-merged.json`）。

### 3.2 全局去重

**Phase 2.5 仅做了批次内去重，Phase 3 需做全局去重**：

1. 将所有批次的 JSON 结果展开为统一的 `问题列表`
2. 按 `文件路径 + 行号 + 维度 + 问题类型` 全局去重（与 Phase 2.5 批次内去重 key 一致）
3. 同一问题多个 Agent 发现时，保留 CRITICAL > WARNING > SUGGESTION
4. 去重后的完整问题列表保留全部字段（文件 / 行号 / 维度 / 类型 / 描述 / 问题代码 / 宕机后果 / 严重程度 / 修复建议）

### 3.3 生成最终报告

使用 Write 工具创建 `{OUTPUT_DIR}/<branch-name>.md`

报告按以下十个部分组织，**每个审计维度独立一章**，章内先按严重程度排序，再按文件分组。同一文件的问题归入同一个文件节点下。全部问题必须经过全局去重后填入。

---

#### 3.3.1 问题统计总览

**按严重程度**：

| 严重程度 | 数量 |
|---|---|
| 🔴 CRITICAL | {n} |
| 🟡 WARNING | {n} |
| 🟢 SUGGESTION | {n} |

**按维度（全局去重后）**：

| 维度 | CRITICAL | WARNING | SUGGESTION | 权重 |
|---|---|---|---|---|
| A. 规范审计 | {n} | {n} | {n} | ×1.5 |
| B. 业务逻辑 | {n} | {n} | {n} | ×1.5 |
| C. 方案合理性 | {n} | {n} | {n} | ×1 |
| D. 性能 | {n} | {n} | {n} | ×1 |
| E. 可靠性（Istio/K8s） | {n} | {n} | {n} | ×2 |
| F. 宕机恢复与逻辑接续 | {n} | {n} | {n} | ×2.5 |
| G. 安全审计 | {n} | {n} | {n} | ×3 |

**涉及文件清单**（出现问题的文件，去重）：

| 文件 | 模块 | 问题数 | 最高严重程度 |
|---|---|---|---|
| {文件路径} | {模块名} | {n} | {CRITICAL/WARNING/SUGGESTION} |

---

#### 3.3.2 A. 规范审计问题（按类分组）

**维度说明**：覆盖命名规范、日志规范、异常规范、事务规范、MyBatis规范、并发规范、i18n规范、包命名规范、枚举规范、循环规范、API交互规范。

**排序规则**：维度内先 CRITICAL → WARNING → SUGGESTION，同一严重程度内按文件分组。

```markdown
### {类名A.java}

| 行号 | 子类型 | 问题描述 | 问题代码 | 修复建议 |
|---|---|---|---|---|
| {n} | {日志规范/异常规范/...} | {描述} | `{问题代码}` | {修复建议} |
| {n} | {...} | {...} | `{...}` | {...} |

### {类名B.java}

### 3.4 审计结论判定标准

根据问题**严重程度 × 维度**交叉判定，而非简单计数：

#### 维度权重（影响越大权重越高）

| 维度 | 权重 | 说明 |
|------|------|------|
| 安全审计（SQL注入/XSS/权限） | ×3 | 直接影响系统安全 |
| 宕机恢复与逻辑接续（状态机/补偿/SAGA） | ×2.5 | 影响故障后业务正确接续，数据丢失风险 |

| 规范审计（异常/日志/并发） | ×1.5 | 代码健壮性 |
| 业务逻辑（边界/null） | ×1.5 | 功能正确性 |
| 性能（分页/批量） | ×1 | 运行时性能 |
| 方案合理性（N+1/重复代码） | ×1 | 长期可维护性 |

#### 判定规则

| 条件 | 结论 | 说明 |
|------|------|------|
| 任何 **CRITICAL × 安全/可靠性** 问题 | **阻塞合并** | 必须修复 |
| 任何 **CRITICAL** 问题 | **需整改** | 影响功能/安全，需修复 |
| 加权 WARNING 分数 ≥ 3 | **需整改** | 多个问题累积 |
| 加权 WARNING 分数 < 3 | **通过** | 仅低影响建议 |

**加权 WARNING 计算示例**：
- 1个安全 WARNING（×3）= 3 → 需整改
- 2个规范 WARNING（×1.5）= 3 → 需整改
- 3个性能建议（SUGGESTION ×0.5）= 1.5 → 通过

#### 维度特殊规则

- **安全维度**：1个 CRITICAL 安全问题直接阻塞，无论其他维度
- **宕机恢复维度**：状态机不完整（无恢复路径）、无补偿机制的跨系统两阶段操作、无法重入的消息处理 → 直接阻塞

- **规范维度**：仅在循环内单条 insert/delete、吞异常（无兜底）时升级为 CRITICAL

## 审计维度

### A. 规范审计
- **MyBatis规范**：
  - XML中禁止使用 `${}` 直接拼接（SQL注入风险）
  - XML中禁止使用 `${@...@}` OGNL表达式调用静态方法
  - XML中禁止使用 `+` 拼接字符串
  - LIMIT/OFFSET必须使用参数绑定，禁止拼接
- **异常规范**：必须使用ErrorCode枚举，禁止RuntimeException/throw new Exception
- **日志规范**：使用占位符`{}`，禁止字符串`+`拼接；debug/info级别日志需条件判断
- **并发规范**：ThreadLocal需在finally中remove；DateTimeFormatter需复用实例
- **事务规范**：@Transactional注解必须在public方法上
- **循环规范**：禁止循环内单条insert/update/delete，必须批量操作
- **i18n规范**：国际化key格式为`dot.case`（如`datakits.framework.cache.exception.xxx`）
- **包命名规范**：使用`dot.case`，遵循`cn.<company>.<product>[.<domain>].<service>[.<layer>][.<business>][.<mvc>]`
- **枚举规范**：枚举类实现BizEnum；messageKey使用国际化KEY；禁止ordinal比较
- **前后端交互**：Long类型序列化为String；时间使用时间戳毫秒；禁止RESTful路径参数

### B. 业务逻辑审计
- 需求覆盖度：代码是否完整实现需求
- 边界值处理：空指针、数组越界、边界数值
- null检查：入参校验、返回值处理
- 数据一致性：事务边界、并发数据一致

### C. 方案合理性审计
- N+1查询：循环内查询数据库
- 重复代码：相同逻辑是否抽取复用
- 过度设计：是否有不必要的抽象

### D. 性能审计
- 分页限制：是否限制最大页数
- 批量操作：是否使用批量insert/update
- 异步超时：@Async方法是否设置超时
- 大数据量：是否考虑流式处理

### E. 可靠性审计

#### E1. 服务宜机能力（Istio + K8s 场景）

> **⚠️ 背景**：系统使用 Istio + Kubernetes 部署，流量治理由 Istio 接管。

- 是否配置了 healthCheck / readinessProbe / livenessProbe（K8s 层）
- 启动探针是否通过（Pod Ready 之前不会被 Istio 路由流量）
- 是否存在启动时预加载数据到内存（Pod 重启后数据丢失）

#### E2. Pod 重启 / 异常恢复（Istio 自动流量切换）

> **核心问题**：Pod 被 Kill 或异常退出后，Istio 会自动将流量切换到其他实例。在这段时间内，**业务逻辑能否无缝接续**？

- **MQ 消息可靠性**：
  - 消费端是否在处理完成后才 ACK（处理失败不确认，消息不丢失）
  - 是否有死信队列（DLQ）兜底消费失败的消息
  - 消费端是否实现幂等（Istio/网络重试可能导致重复投递）
- **定时任务可靠性**：
  - 定时任务是否持久化执行进度（任务状态外化到 DB，支持中断后恢复）
  - 多实例部署时是否使用分布式锁防重复执行（ShedLock 或 Redis 锁）
  - 任务失败是否有补偿或重试机制
- **缓存可靠性**：
  - 是否存在缓存击穿（热点 key 过期瞬间大量请求击穿到 DB）
  - 缓存故障时是否有降级兜底（走数据库）
  - 缓存数据是否可接受丢失（有无降级路径）
- **分布式锁可靠性**：
  - 锁是否设置了合理的超时时间（防止 Pod 被 Kill 后锁永不过期）
  - 锁是否绑定唯一标识（防止误删他人持有的锁）

#### E3. 异常恢复与容错（基于 Istio 流量治理）

> **⚠️ 背景**：系统使用 Istio 作为服务网格，熔断、重试、超时、限流等流量治理能力由 Istio 统一配置。应用层审计聚焦于 **Pod 重启/异常时业务逻辑能否继续**，即 Istio 无法覆盖的部分。

- **幂等性保障**：
  - MQ 消费端是否实现幂等（Istio 重试可能导致重复投递）
  - 对外 HTTP/Feign 调用是否保证幂等（Istio 重试不会导致数据重复）
  - 重试场景下数据库操作是否有唯一约束或状态机校验
- **降级兜底逻辑**：
  - 远程服务不可用时，是否返回兜底数据（而非直接抛异常）
  - 非核心依赖故障时，核心业务是否可降级继续
- **状态外部化**：
  - 业务状态是否存储在 DB/Redis，而非内存（Pod 重启后状态不丢失）
  - 定时任务是否持久化执行进度（任务状态外化，支持中断恢复）
- **数据一致性**：
  - 长事务是否拆分（单次请求内完成，避免跨请求状态不一致）
  - 分布式事务是否使用 Seata TCC 模式（幂等 + 空回滚 + 悬挂防护）
  - 补偿事务/消息表是否用于最终一致性场景
- **资源泄漏防护**：
  - DB 连接是否使用 try-with-resources 或 finally 释放
  - 线程池是否在 Bean 初始化时创建而非每次请求创建
  - 超时未返回的异步任务是否设置了最大生命周期
- **并发控制**：
  - 多实例部署时，定时任务/全局操作是否使用分布式锁
  - 分布式锁是否设置超时（防止 Pod 被 Kill 后锁无法释放）

#### E4. 数据一致性
- **事务策略**：
  - 长事务是否拆分为短事务
  - 分布式事务是否使用 XA / TCC / SAGA 模式
  - 事务回滚是否完整（finally 中清理资源）
- **数据变更兜底**：
  - 是否有改名字段需要撤销操作脚本
  - 危险操作（批量删除/更新）是否有人工确认或二次校验

#### E5. 兜底与防护完整性
- **禁止吞没异常**：catch 块不能为空或只打印日志不处理
- **资源释放**：IO / DB 连接 / 线程池是否使用 try-with-resources 或 finally 释放
- **日志中断点**：关键操作是否有日志记录（方便故障后恢复追踪）
- **幂等性保障**：MQ 消费/重试/并发场景下是否保证幂等

### F. 宕机恢复与逻辑接续审计（重点维度）

> **⚠️ 核心问题**：JVM在处理业务逻辑的过程中宕机，重启后业务能否正确接续？这类问题无法通过静态代码风格检查发现，必须通过**故障模式分析（Failure Mode Analysis）**来审计。

#### F1. 状态机完整性审计

**核心问题**：对每个状态变更节点，问"如果在这行代码之后宕机，会发生什么？"

审计要点：
- 状态流转是否形成闭环（每个非终态都应有恢复路径）
- 状态变更是否原子化（避免中间状态导致数据不一致）
- 是否有"卡死"状态（长期停留在某个状态，没有超时机制）

典型问题模式：
```
// 问题代码示例
task.setStatus(PROCESSING);   // 宕机点1：状态已变更
exportTaskRepository.updateById(task);
sendKafkaMessage();            // 宕机点2：Kafka发送失败或发送成功但应用已宕机

// 如果在"状态变更之后、Kafka发送之前"宕机：
// - 消息不会重发（因为消费端可能已处理或未发送）
// - 任务永远卡在PROCESSING状态
```

审计检查项：
1. 状态变更前是否有checklist（防止重复处理）
2. 状态变更和下游操作的顺序是否正确（先持久化还是先发消息？）
3. 是否存在"两步操作但只检查一个状态"的情况
4. 是否有超时机制将"卡死"状态恢复

#### F2. 可重入性审计（幂等消息处理）

**核心问题**：消息被重复投递（Kafka重试、网络抖动、消费者重启），业务逻辑能否正确处理？

审计要点：
- 状态检查是否接受所有"合法重入"的状态
- 是否实现了"检查-执行-再检查"的幂等模式
- 是否存在"状态已被更新但后续操作失败导致不可重入"的情况

典型问题模式：
```
// 问题代码示例
if (!PENDING.equals(task.getStatus())) {
    return;  // 幂等保护 ✓
}
// 但如果在这之后宕机：
task.setStatus(PROCESSING);
exportTaskRepository.updateById(task);  // 状态已变
// 后续操作失败，重启后消息重推
// 此时状态已不是PENDING，消息被跳过，任务丢失 ✗
```

审计检查项：
1. 状态检查是否同时验证"当前持有者"（workerId/instanceId）
2. PROCESSING状态是否设置了最大处理时长（超时则允许其他实例接管）
3. 是否存在"状态被更新但未完成，导致无法重入"的情况

#### F3. 补偿机制审计

**核心问题**：跨系统两阶段操作（如DB+MQ、DB+OSS、MQ+外部服务），第二步失败后是否有补偿？

典型问题模式：

**模式1：DB更新成功，MQ发送失败**
```java
// 问题代码示例
task.setStatus(SUCCESS);
exportTaskRepository.updateById(task);  // 成功
streamBridge.send("export-out", message); // 失败被catch，只打日志
// 任务状态已为SUCCESS，但下游消费者永远收不到消息 ✗
```

**模式2：DB状态更新，外部服务调用失败**
```java
// 问题代码示例
task.setStatus(UPLOADED);
exportTaskRepository.updateById(task);  // 成功
fileCenter.notifyMerge(fileId);           // 失败，抛异常
// 状态已是UPLOADED，但文件中心没有执行合并
// 后续补偿任务看到UPLOADED状态，认为已完成，不会重试 ✗
```

**模式3：分片上传成功，合并通知失败**
```java
// 问题代码示例
uploadParts();                    // 分片全部上传成功
fileCenter.notifyMerge(fileId);   // 合并通知失败
fileCenter.cancelUpload(fileId);   // 取消失败，只打日志
// OSS上已有分片文件，但合并失败
// 产生大量OSS垃圾碎片，无人清理 ✗
```

审计检查项：
1. 两阶段操作是否有补偿事务（SAGA模式或消息表）
2. 失败后的补偿操作是否有幂等保护
3. 补偿失败是否有告警（不能静默失败）
4. 是否存在"资源泄漏"风险（OSS分片、临时文件等）

#### F4. 分布式事务边界审计

**核心问题**：跨数据库/跨服务的操作，是否有正确的事务边界？

审计要点：
- 事务是否包含了所有需要原子化的操作
- 是否存在"长事务"（跨网络调用在事务内）
- 是否有"在事务中调用外部服务"的情况

典型问题模式：
```java
// 问题代码示例
@Transactional
public void submitTask() {
    exportTaskRepository.save(task);        // DB操作 ✓
    streamBridge.send("export-out", msg);   // 网络调用 ✗ 在事务中调用外部服务
    // 如果事务超时，DB已提交但MQ未发送
    // 或者MQ发送成功但DB回滚
}
```

审计检查项：
1. 事务是否只包含DB操作（不包括网络调用）
2. 跨服务操作是否使用了最终一致性方案（消息表+补偿任务）
3. 是否存在"本地事务成功、外部调用失败"的场景

#### F5. 宕机恢复与补偿任务审计

**核心问题**：是否有定时任务扫描"卡死"状态并恢复？

审计要点：
- 是否有补偿任务定期扫描超时未完成的任务
- 补偿任务是否会重复执行导致数据问题
- 补偿任务是否有防重复执行的保护

审计检查项：
1. PROCESSING/中间状态是否有最大处理时长限制
2. 是否有补偿/恢复定时任务
3. 补偿任务是否使用了分布式锁
4. 补偿任务的恢复策略是否正确（是否能正确接续）

#### F6. 故障模式分析（FMEA）追问清单

对每个关键的业务文件，审计时必须追问以下问题：

**消息消费类**（Listener、Consumer）：
1. 消息被重复投递时，状态检查是否会跳过？（是否支持重入）
2. 如果在"更新状态为PROCESSING"之后宕机，重启后消息重推会被跳过吗？
3. ACK是在处理成功之后还是之前？
4. 失败消息是否有DLQ兜底？

**导出/异步任务类**（ExportService、AsyncService）：
1. 任务状态变更和Kafka消息发送的顺序是什么？
2. 如果在"更新数据库"之后、"发送Kafka"之前宕机怎么办？
3. 文件上传到OSS成功后，通知外部系统失败如何处理？
4. 是否有补偿定时任务扫描"卡死"状态？

**分片上传类**：
1. 分片上传成功但合并通知失败，会产生OSS垃圾碎片吗？
2. 有没有清理OSS垃圾的补偿机制？
3. 取消上传失败后有没有告警？

**多阶段状态机类**：
1. 每个状态变更是否有超时机制？
2. 超时后状态如何恢复？
3. 恢复后能否正确接续？

### G. 安全审计
- **注入攻击**：SQL注入、XSS、命令注入
- **敏感信息**：硬编码密码/密钥/Token/AWS凭证
- **权限校验**：接口是否缺少权限检查
- **越权风险**：数据访问是否校验所属权
- **日志脱敏**：日志是否打印敏感字段（手机号、身份证、密码）

### 严重程度判定标准（各维度通用）

| 严重程度 | 判定规则 |
|---|---|
| **CRITICAL** | 违反规范，且在特定运行时场景下**必然**导致功能错误、数据丢失或安全问题 |
| **WARNING** | 违反规范，在常见运行时场景下**可能**导致问题，或存在一定风险 |
| **SUGGESTION** | 违反规范，但当前上下文风险极低，属于优化建议 |

> **同一问题可能因上下文不同而判不同级别**。例如：日志字符串拼接在核心业务流程中判 WARNING，在简单的工具类中判 SUGGESTION。

1. **读取规范文件**：使用 Read 工具读取上述 6 个规范文件（相对路径），理解审计标准
2. **读取计划文件**：使用 Read 工具读取本计划文件，了解本次批次的文件列表和审计范围
3. **读取文件内容**：使用 Read 工具逐一读取每个需要审计的 Java/XML 文件的完整内容
   > **必须读取文件内容后才能进行分析，不能仅靠 git diff 的变更片段判断问题**
   > git diff 只显示变更的部分，完整判断需要结合文件的上下文
4. **执行 git diff**：获取代码变更对比，了解本次提交改了什么：
   ```bash
   git diff ${BASE_BRANCH}...${BRANCH} -- {文件路径1} {文件路径2} ...
   ```
   > ⚠️ 使用 `${BRANCH}` 而非 `HEAD`，确保多批次执行时取到正确分支差异。
5. **按审计维度（A-F）逐文件分析**：
   - 对每个文件，结合文件内容和 diff，逐一检查是否符合规范
   - 检查每个问题必须包含：文件路径、行号、严重程度、维度、类型、描述、问题代码、修复建议
6. **写入结果文件**：将结果按指定 JSON 格式写入结果文件

## 关键改进说明

### 改进1：Auto Mode 兼容

- 开头统一判断运行模式
- 所有确认点条件分支：手动模式等待确认，Auto Mode自动执行
- Auto Mode 下批次间自动流转，不中断

### 改进2：分步骤确认

- 主计划包含明确的执行步骤
- 每个步骤完成后向用户确认（手动模式）
- 防止大模型遗漏任务

### 改进3：动态批次执行

- 不限制总agent数量
- 每批次最多4个agent并行
- 根据文件数量动态分组

### 改进4：动态批次ID

- 批次ID用数字序号，避免路径冲突
- 批次名附加到文件内容中，不作为文件名

### 改进5：动态更新待合并列表

- 每批次完成后更新主计划
- 结果文件路径追加到列表中
- 最终汇总时读取所有文件

### 改进6：主计划包含所有结果

- 所有审计结果先汇总到主计划
- 最终生成清晰的审计报告
- 便于追溯和复查

### 改进7：中断恢复与保障治理

- 支持三个级别的中断恢复：新审计 / 不完整继续 / 特定批次中断
- 递阶持久化避免数据丢失：批次级别 / 三段处理程序 / Agent 检查点
- Agent 可被中断后自动恢复：每 5 文件一份中间结果保存
- 特定失败有备份 Agent 重试

### 改进8：审计结论判定标准

- 明确的判定规则：CRITICAL → 阻塞，3+ WARNING → 需整改，否则通过
- 避免主观判断，保持一致性

### 改进9：安全审计维度

- 新增 F. 安全审计维度
- 覆盖注入攻击、敏感信息、权限校验、日志脱敏

### 改进10：超时动态计算

- 超时时间 = max(5, min(文件数/2, 20)) 分钟
- 避免大文件超时或小文件等待过久
